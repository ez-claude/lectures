# Claude Code 마스터 설정 > 이 파일은 항상 로드됨. > 세부 규칙은 rules/ 폴더 참조.

이 파일 자체가 뭔가요?

그냥 텍스트 파일입니다. 프로젝트 폴더에 넣어두면 AI가 대화를 시작할 때 자동으로 읽고 모든 작업에 적용합니다. 별도 설치나 설정은 필요 없습니다.

도구에 따라 파일명만 다릅니다: Claude Code는 CLAUDE.md, Antigravity는 프로젝트 루트에 RULES.md, Cursor는 .cursorrules.

## 절대 규칙 (최우선) 1. 이모지 금지 - 텍스트로 대체 ([OK], [FAIL], [Step]) 2. 한글 응답 - 모든 출력은 한글 3. 버전 잠금 - state/versions.json 우선 4. 보안 필수 - API 키 환경변수만, 하드코딩 금지 5. Pillar 시스템 필수 - Build Mode에서 Step 0~5 강제 실행 6. 기록 전 검증 - 도구로 확인 가능한 것은 도구로 확인 후 기록
수정 가능

절대 규칙 = AI의 헌법

AI가 매 대화마다 반드시 지키는 기본 행동입니다. 이게 없으면 AI가 영어로 답변하거나, API 키를 코드에 직접 써버립니다.

본인 프로젝트에 맞게 규칙을 추가하거나 수정해도 됩니다. 예를 들어 영어 프로젝트면 2번을 "영어 응답"으로 바꾸세요.

## [GATE SYSTEM] 코드 생성 전 필수 관문 > "만들어줘", "구현해줘", "코드" 키워드 감지 시 자동 실행. > 관문 통과 전 코드 생성 금지 ### GATE -1: Pre-Plan Gate 플랜 품질 검증. 5개 체크: 경쟁분석 / 근본원인 / 품질기준 / 도메인규칙 / 검증방법 → 3개 미만 시 차단 ### GATE 0: 버전 파일 Read Read: versions.json → [GATE 0 통과] 라이브러리/AI 모델 정확한 버전 확인 ### GATE 1: 패턴 파일 Read Read: patterns.json → [GATE 1 통과] 이전 실수 패턴 확인 ### GATE 2 관련 패턴 경고 → [GATE 통과] 출력 후 코드 생성 시작
그대로 두세요

GATE = 코드 짜기 전 검문소

"만들어줘"라고 하면 AI가 바로 코드를 짜지 않습니다. 먼저 사용 중인 라이브러리 버전을 확인하고, 이전에 같은 실수를 한 적이 있는지 체크합니다.

이게 없으면 AI가 존재하지 않는 라이브러리 버전이나 삭제된 함수를 쓰는 실수를 합니다. GATE -1은 플랜 자체의 품질도 검증합니다. 경쟁 분석이나 검증 방법이 빠져있으면 플랜 단계에서 차단합니다.

## [POST-BUILD] 코드 생성 후 필수 검증 코드 생성 완료 후 반드시: [A] versions.json 일치: [OK/FAIL] [B] patterns.json 위반: [OK/FAIL] [C] API 키 하드코딩: [OK/FAIL] [D] import 누락: [OK/FAIL]
그대로 두세요

만든 후에도 자동 4가지 검증

코드를 다 만들고 나서도 끝이 아닙니다. 버전이 맞는지, 이전 실수를 반복하지 않았는지, API 키가 노출되지 않았는지, import가 빠지지 않았는지 자동으로 체크합니다. 문제가 있으면 AI가 스스로 수정합니다.

## [DEBUGGING] 근본 원인 분석 > 스킬 `systematic-debugging` 기반. > 트리거 키워드 감지 시 필수 실행. ### 트리거 키워드 "아직도", "똑같아", "그대론데", "달라진게 없는데", "여전히", "안 바뀌었어", "같은 에러", "또" ### 필수 실행 순서 [1/5] 문제 유형 식별 [2/5] 에러 로그/콘솔 실제 데이터 수집 [3/5] 데이터 기반 원인 분석 (추측 금지!) [4/5] 수정 적용 [5/5] 사용자에게 보고: 원인 + 근거 + 해결 방안 ### 강제 규칙 - [금지] 데이터 확인 없이 코드 수정 - [금지] "아마도", "~인 것 같습니다" 추측성 표현 - [필수] 도구 실행 결과 → 원인 분석 → 수정 제안
그대로 두세요

도구(스킬): systematic-debugging

이게 "도구(스킬)"의 실제 작동 예시입니다. "아직도 안 돼"라고 말하면 AI가 추측 모드를 끄고 증거 기반 분석 모드를 켭니다.

"아마 이거 같아요" 식의 추측을 금지하고, 반드시 에러 로그를 먼저 확인한 뒤 원인을 찾습니다. systematic-debugging 스킬이 설치되어 있어서 디버깅 품질이 완전히 달라집니다.

스킬 = 버튼만 누르면 결과가 나오는 도구라고 했죠? 여러분은 "아직도 안 돼"라고 말만 하면 됩니다.

## [COMMIT] 검증 후 커밋 > 스킬 `verification-before-completion` > + `finishing-a-development-branch` 기반 "완료", "배포", "커밋" 감지 시: 1. 보안 스캔 실행 (API 키 노출 검사) 2. 코드 품질 검증 (lint, 타입 체크) 3. FAIL → 커밋 중단, 통과 시 git commit
그대로 두세요

도구(스킬) 2개가 합쳐진 예시

verification-before-completion = 완료 전에 자동으로 검증하는 도구
finishing-a-development-branch = 커밋 전에 보안+품질을 체크하는 도구

이 두 스킬이 합쳐져서 "커밋해줘" 한마디에 검증 → 통과 확인 → 커밋까지 전부 자동으로 진행됩니다. 스킬은 이렇게 여러 개를 조합해서 쓸 수도 있습니다.

## 세션 모드 시스템 | 모드 | 트리거 | 동작 | |----------------|-------------------|---------------------------| | Plan-Light | "~할까?", 질문형 | 가벼운 분석, 선택지 제시 | | Plan-Deep | "분석해줘" | 심층 분석 + 경쟁 제품 비교 | | Build | "만들어줘", 명령형 | GATE 발동 → 코드 생성 |

말투에 따라 AI 모드가 바뀝니다

"~할까?"라고 물어보면 가볍게 분석하고 선택지를 줍니다. "만들어줘"라고 하면 위에서 본 GATE 시스템이 발동됩니다. 같은 AI인데 질문 방식에 따라 동작이 완전히 달라지는 것이 규칙 파일의 힘입니다.

## React 작업 자동 감지 > 스킬 `vercel-react-best-practices` 기반 React/Next.js 키워드 감지 시: → 스킬의 CRITICAL 8 규칙 자동 적용 (Defer Await, Suspense, Barrel Import 회피 등)
그대로 두세요

도구(스킬): React 자동 최적화

React 관련 코드를 쓸 때 AI가 자동으로 감지하고 성능 최적화 8가지 규칙을 적용합니다. 여러분이 "React 최적화해줘"라고 말할 필요도 없습니다. 키워드만 감지되면 스킬이 알아서 발동됩니다.

## 필수 보안 규칙 ### Git Push 전 검증 (필수!) grep -rE "(AIzaSy|sk-|ghp_|gho_)" . | 금지 패턴 | 올바른 방식 | |---------------------|--------------------------------| | AIzaSy... (Google) | os.getenv('GOOGLE_API_KEY') | | sk-... (OpenAI) | os.getenv('OPENAI_KEY') | | ghp_... (GitHub) | os.getenv('GITHUB_TOKEN') |
그대로 두세요

API 키 유출 = 수백만 원 과금

실제로 GitHub에 API 키를 실수로 올려서 수백만 원이 과금된 사례가 많습니다. 이 규칙이 있으면 AI가 코드를 만들 때 자동으로 .env 파일 방식을 사용합니다. 실수로 키를 코드에 직접 쓰면 AI가 즉시 경고합니다.

## 코드 리뷰 > 스킬 `requesting-code-review` > + `receiving-code-review` 기반 핵심 원칙: 전체 검토 후 한꺼번에 보고 ("일단 이것만" 금지) 1. 입력 확인 → 2. 출력 확인 3. 구조 매칭 → 4. 전체 흐름
그대로 두세요

도구(스킬): 코드 리뷰 자동화

"검토해줘" 한마디면 입력→출력→전체 흐름을 한 번에 검토합니다. "일단 이것만 볼게요"라고 대충 넘어가는 걸 금지시켜놨기 때문에, 모든 문제를 한 번에 찾아서 보고합니다.

## 에이전트 부서 라우팅 > 스킬 `dispatching-parallel-agents` > + `subagent-driven-development` 기반 > 상세 설정: routing.json | 부서 | 에이전트 | 트리거 | |------|----------------------------------|----------------| | 검증 | pillar-gate | 검증/리뷰 | | | code-reviewer | /보안 | | | security-scanner | /디버깅 | | | pattern-debugger | | | 개발 | python-backend | Python/React | | | frontend-react | /n8n/구현 | | | n8n-builder | | | 설계 | architect | 설계/아키텍처 | 3개 이상 파일 변경 시 토론 패턴 자동 발동 (architect + code-reviewer 병렬)
그대로 두세요

핵심: "직원(서브에이전트)" 부서

이 표 하나가 AI를 팀으로 만듭니다.

검증부 4명 - code-reviewer(코드 리뷰), security-scanner(보안 검사), pattern-debugger(패턴 분석), pillar-gate(품질 관문)

개발부 3명 - frontend-react(화면 개발), python-backend(서버 개발), n8n-builder(자동화)

설계부 1명 - architect(전체 구조 설계)

파일이 3개 이상 바뀌면, 설계팀이 먼저 구조를 잡고 → 개발팀이 코딩하고 → 검증팀이 리뷰합니다. 앞에서 본 스킬이 "도구"라면, 이 에이전트들은 "직원"입니다. 텍스트를 넣어둔 것뿐인데 AI 행동이 완전히 달라집니다.

AI가 "다 했어요" 할 때의 기준

AI가 작업 완료를 선언하려면 이 4가지를 모두 통과해야 합니다. 보안 스캔 통과 + 빌드 경고 0개 + PR 링크 제공까지. "대충 했어요"가 불가능한 구조입니다.
