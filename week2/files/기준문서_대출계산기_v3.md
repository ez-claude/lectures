# 대출이자 계산기 -- 개발 기준서

> 이 문서를 AI에게 주고 "시작해줘"라고 하면 된다.
> AI는 이 기준서를 읽고, 4단계의 설계 문서를 스스로 만들고, 그 문서를 기반으로 구현까지 완료한다.

---

## 0. 실행 환경 (Antigravity)

```
/turbo-all
```

- 위 명령을 프로젝트 시작 시 먼저 실행한다
- 모든 권한 요청(Allow, Trust, 파일 수정 등)을 자동 승인한다
- 사용자에게 중간 확인을 묻지 않고 끝까지 자율 실행한다
- npm install, npx, 외부 패키지 설치 명령은 절대 실행하지 않는다
- 순수 HTML/CSS/JavaScript만 사용한다 (프레임워크 금지)

### 필수 스킬

아래 스킬이 사용 가능하면 해당 단계에서 실행한다.
사용할 수 없으면 "대체 방법"을 따른다.

| 스킬 | 사용 단계 | 기능 | 대체 방법 |
|------|----------|------|----------|
| `/design-research` | Step 2 | 최신 UI 트렌드 검색, 디자인 앵커 선택, design-decisions.md 생성 | WebSearch로 직접 검색 |
| `/frontend-design` | Step 4 | AI 기본 디자인 탈피, 전문가 수준 프론트엔드 구현 | Step 2 디자인 결정서 기반으로 직접 구현 |
| `/web-design-guidelines` | 검증 | Vercel 웹 인터페이스 가이드라인 자동 검증 | 접근성/반응형 수동 확인 |
| `/webapp-testing` | 검증 | Playwright 기반 웹앱 자동 테스트 | 브라우저에서 수동 검증 |

**스킬 미설치 시 설치 방법:**

스킬은 `.claude/skills/[스킬명]/SKILL.md` 파일로 설치된다.
해당 경로에 파일이 없으면 아래 순서로 처리하라:

1. `.claude/skills/` 폴더 확인 — 이미 설치되어 있는지 확인
2. 없으면 WebSearch로 `"claude code [스킬명] skill SKILL.md"` 검색하여 내용 확보
3. 검색도 실패하면 "대체 방법"으로 수동 진행

## 1. 프로젝트 목표

대출을 고려 중인 일반인(금융 비전문가)이 **은행 방문 없이 대출 조건을 비교하고 판단**할 수 있는 계산기를 만든다.

핵심 기준:
- 은행 앱보다 빠르게 결과를 볼 수 있어야 한다
- 네이버 대출계산기보다 시각적으로 우수해야 한다
- 엑셀보다 직관적이어야 한다
- 계산 결과는 실제 은행 계산과 1원 이내 오차여야 한다

## 2. 사용자 시나리오

이 시나리오들을 분석하고, 각 시나리오를 해결하는 기능을 설계하라.

### 시나리오 A: 첫 대출
> "전세 대출 받으려는데, 월 얼마나 나가는지 모르겠어요. 3억 빌리면 월급에서 감당이 되나요?"

사용자가 알고 싶은 것: 월 납입금, 내 소득 대비 부담 수준

### 시나리오 B: 조건 비교
> "A은행은 3.5% 30년, B은행은 4.0% 20년인데 뭐가 유리한지 모르겠어요."

사용자가 알고 싶은 것: 두 조건의 총 이자 차이, 월 납입금 차이

### 시나리오 C: 상환 방식 고민
> "원리금균등이랑 원금균등이 뭔지 알겠는데, 실제로 얼마나 차이 나는지 모르겠어요."

사용자가 알고 싶은 것: 두 방식의 총 이자 차이, 시간에 따른 납입금 변화

### 시나리오 D: 조기상환
> "5년 후에 목돈이 생기면 일부 갚을 수 있는데, 그러면 이자가 얼마나 줄어요?"

사용자가 알고 싶은 것: 조기상환 시 절약되는 이자, 남은 기간 변화

## 3. AI 실행 규칙 (핵심)

> **코드를 짜기 전에, 아래 4개의 설계 문서를 먼저 파일로 작성하라.**
> 각 문서를 충분히 깊게 작성한 후, 그 문서들을 기반으로 구현한다.
> 문서 작성 없이 바로 코드를 짜는 것을 금지한다.

### Step 1: 사용자 심층 분석서 작성 (`docs/01_사용자분석.md`)

아래 질문에 하나하나 깊게 답하라. 한두 줄로 넘어가지 말 것.

**사용자가 진짜 원하는 것:**
- 이 앱을 여는 순간 사용자의 감정 상태는? (불안? 급함? 호기심?)
- 표면적으로 원하는 것(월 납입금 계산) 말고, 진짜 원하는 것은?
- 여기 오기 전에 뭘 하고 있었고, 여기서 나간 후 뭘 할까?
- 한 번 쓰고 끝낼까, 며칠/몇 주에 걸쳐 다시 올까?
- 결과를 혼자 볼까, 배우자/부모/은행 직원에게 보여줄까?
- 이 앱에서 원하는 걸 못 얻으면 다음 행동은?

**초보자가 막힐 수 있는 지점:**
- "원리금균등"이라는 단어를 처음 보는 사람은 어떤 반응을 할까?
- 입력 필드에서 "이게 뭘 넣으라는 건지 모르겠다"가 발생할 수 있는 곳은?
- 결과 화면에서 숫자가 많을 때 "뭘 봐야 하는지 모르겠다"가 발생하면?
- 실수로 잘못 입력했을 때 "어떻게 되돌리지?"가 발생하면?

**이 문서에 적힌 시나리오 너머:**
- 시나리오 A~D 외에, 이 앱을 쓸 만한 다른 상황이 있을까?
- 경쟁 서비스(네이버 대출계산기, 은행 앱)를 써본 사용자가 가장 불만인 점은?

### Step 2: 디자인 리서치 + 기준서 작성 (`docs/02_디자인기준.md`)

**이 단계를 건너뛰고 코딩을 시작하면 안 된다.**
디자인 결정을 먼저 문서화한 후에만 Step 3으로 넘어갈 수 있다.

**디자인 리서치 실행:**

`/design-research` 스킬을 실행하라 (도메인: "대출 계산기").
스킬이 자동으로 최신 UI 검색 → 앵커 선택 → design-decisions.md 생성까지 처리한다.

스킬을 사용할 수 없으면 WebSearch로 직접 리서치하라:
- "loan calculator UI design 2025 2026"
- "fintech app dashboard modern design"
- "best calculator web app design inspiration"

리서치 후 `docs/design-decisions.md`에 기록:
- 참고한 서비스/사이트 (최소 2개, URL 포함)
- 컬러 팔레트: 메인, 서브, 배경, 강조 (hex 코드)
- 선택 이유 (1~2줄)
- 폰트, 컴포넌트 스타일 방향

**첫인상 설계:**
- 사용자가 앱을 열었을 때 첫 3초 안에 느껴야 할 감정은?
- "이거 은행 앱이야? 진짜 서비스야?"라는 반응이 나오려면 어떤 요소가 필요한가?

**정보 구조 설계:**
- 숫자가 많은 화면에서 사용자가 길을 잃지 않으려면? (그룹핑, 단계 구분, 시각적 위계)
- 가장 중요한 숫자(월 납입금)를 어떻게 강조할 것인가?
- 부가 정보(상환 스케줄, 차트)는 어떤 순서로 보여줄 것인가?

**초보자 배려 설계:**
- 전문 용어(원리금균등, DTI)를 모르는 사람을 위한 설명 방식은?
- 각 입력 필드에 기본값과 안내를 어떻게 넣을 것인가?
- 모바일에서 엄지손가락 하나로 모든 조작이 가능한가?

### Step 3: 기능 + 기술 설계서 작성 (`docs/03_기능설계.md`)

Step 1, 2를 기반으로, 구현할 기능과 기술적 주의사항을 정리하라.

**기능 목록:**
- 시나리오 A~D를 해결하는 기능
- Step 1에서 발견한 추가 니즈를 해결하는 기능
- 각 기능의 우선순위와 이유

**계산 정확성 (은행 기준 — 반드시 지켜라):**

| 규칙 | 내용 | 이유 |
|------|------|------|
| 이자 절사 | 원 미만 버림 (Math.floor). 반올림 금지 | 은행 실무 기준 |
| 총 상환액 | 각 회차 납입금을 하나하나 더해서 합산 | 절사 누적 오차 반영 |
| 마지막 회차 | 잔액이 0이 되도록 보정 | 누적 오차 정리 |

**AI가 흔히 놓치는 것들 (반드시 확인):**
- 금액 입력 시 천 단위 콤마 자동 포맷팅 + 콤마 포함 입력 허용
- 슬라이더와 입력 필드가 항상 동기화되는가?
- 0원, 0%, 0년 입력 시 에러 없이 처리되는가?
- 40년(480회차) 계산 시 렌더링이 느려지지 않는가?
- 새로고침하면 입력값이 날아가는가? (날아가면 안 됨)

**최신 웹 기술 리서치 (권장):**

구현 전에 WebSearch로 최신 CSS/JS 기능을 검색하여 적용 가능한 것이 있는지 확인하라:
- "modern CSS features 2025 2026" — container queries, :has(), color-mix() 등
- "modern JavaScript features 2025" — 최신 문법, 새 API
- "web API new features 2025" — View Transitions, Popover API, Dialog 등

적용 가능한 최신 기능이 있으면 `docs/03_기능설계.md`에 기록하고 구현에 반영하라.
단, 브라우저 호환성을 확인하고 (Chrome/Edge/Safari 최신 버전 기준) 안정적인 기능만 사용하라.

### Step 4: 위 3개 문서를 읽고 구현하라

- 3개 문서를 **전부 읽은 후** 코드를 작성하라
- `/frontend-design` 스킬이 있으면 실행하여 전문가 수준 프론트엔드를 구현하라
- 문서에 적은 내용이 코드에 빠짐없이 반영되었는지 확인하라
- 한 파일이 500줄을 넘지 않도록 분리하라 (HTML/CSS/JS)

## 4. 구현 기준

- **형태**: 웹앱. 서버/DB 없이 브라우저에서만 동작
- **언어**: 전체 UI 한글. 영어 메뉴/라벨 금지
- **첫 화면**: 빈 화면 금지. 기본값이 채워져 있어서 바로 결과가 보여야 한다
- **디자인 수준**: Step 2에서 리서치한 결과를 기반으로 금융 앱다운 신뢰감 있는 UI
- **반응형**: 데스크탑과 모바일 모두 자연스럽게 동작

## 5. 금지사항

| 금지 | 이유 |
|------|------|
| 서버, 데이터베이스 사용 | 브라우저만으로 동작해야 함 |
| 로그인, 회원가입, 개인정보 수집 | 불필요 |
| 외부 API 호출 | 오프라인에서도 동작 |
| 외부 이미지 URL | 로딩 실패 위험 |
| npm install, 외부 패키지 설치 | Antigravity 환경 제약 |
| 영어 UI | 사용자가 한국인 비전문가 |

## 6. 검증

구현이 끝나면 아래를 스스로 검증하라:

1. **시나리오 A~D가 모두 해결되는가?** 각 시나리오의 사용자가 원하는 답을 얻을 수 있는지 확인
2. **계산 정확성**: 1억원 / 4.5% / 30년 / 원리금균등 → 월납입금 약 506,685원 (+-1원)
3. **설계 문서 반영**: Step 1~3에서 작성한 내용이 코드에 빠짐없이 반영되었는가?
4. **초보자 테스트**: "원리금균등"이 뭔지 모르는 사람이 이 앱을 쓸 수 있는가?
5. **모바일**: 375px 화면에서 모든 기능이 동작하는지
6. **빈틈**: 사용자가 "이것도 되면 좋겠는데"라고 할 만한 기능이 빠져있지 않은지 스스로 점검하고, 있으면 추가
