# [프로젝트명] — 개발 기준서

> 이 문서를 AI에게 주고 "시작해줘"라고 하면 된다.
> AI는 이 기준서를 읽고, 설계 문서를 스스로 만들고, 그 문서를 기반으로 구현까지 완료한다.
>
> **사용법**: 대괄호 `[...]` 안의 내용을 실제 프로젝트에 맞게 교체하세요.

---

## 0. 실행 환경 (Antigravity)

```
/turbo-all
```

- 위 명령을 프로젝트 시작 시 먼저 실행한다
- 모든 권한 요청(Allow, Trust, 파일 수정 등)을 자동 승인한다
- 사용자에게 중간 확인을 묻지 않고 끝까지 자율 실행한다
- npm install, npx, 외부 패키지 설치 명령은 절대 실행하지 않는다
- 순수 HTML/CSS/JavaScript만 사용한다 (프레임워크 금지)

### 필수 스킬

아래 스킬이 사용 가능하면 해당 단계에서 실행한다.
사용할 수 없으면 "대체 방법"을 따른다.

| 스킬 | 사용 단계 | 기능 | 대체 방법 |
|------|----------|------|----------|
| `/design-research` | Step 2 | 최신 UI 트렌드 검색, 디자인 앵커 선택, design-decisions.md 생성 | WebSearch로 직접 검색 |
| `/frontend-design` | Step 4 | AI 기본 디자인 탈피, 전문가 수준 프론트엔드 구현 | Step 2 디자인 결정서 기반으로 직접 구현 |
| `/web-design-guidelines` | 검증 | Vercel 웹 인터페이스 가이드라인 자동 검증 | 접근성/반응형 수동 확인 |
| `/webapp-testing` | 검증 | Playwright 기반 웹앱 자동 테스트 | 브라우저에서 수동 검증 |

**스킬 미설치 시 설치 방법:**

스킬은 `.claude/skills/[스킬명]/SKILL.md` 파일로 설치된다.
해당 경로에 파일이 없으면 아래 순서로 처리하라:

1. `.claude/skills/` 폴더 확인 — 이미 설치되어 있는지 확인
2. 없으면 WebSearch로 `"claude code [스킬명] skill SKILL.md"` 검색하여 내용 확보
3. 검색도 실패하면 "대체 방법"으로 수동 진행

## 1. 프로젝트 목표

[대상 사용자]가 **[기존 방법] 없이 [핵심 행동]을 [시간] 안에 완료**할 수 있는 도구를 만든다.

> 예시: "소규모 사업자가 엑셀 없이 전문적인 견적서를 3분 안에 작성하고 바로 출력할 수 있는 도구"

핵심 기준:
- [기존 도구]보다 빠르고 직관적이어야 한다
- [핵심 결과물]이 실제로 쓸 수 있을 만큼 깔끔해야 한다
- [핵심 로직 — 예: 계산, 변환, 정렬]은 정확해야 한다
- 디자인은 [벤치마크 서비스 — 예: 토스, 카카오뱅크, 노션] 수준이어야 한다

## 3. AI 실행 순서 (코드 작성 전 필수)

코드를 바로 작성하지 마라. 아래 4단계를 순서대로 실행하라.

### Step 1: 사용자 분석서 작성 (`docs/01_사용자분석.md`)

위 시나리오를 바탕으로, 아래에 대해 깊이 생각하고 정리하라.

**사용자가 진짜 원하는 것:**
- 이 앱을 여는 순간 사용자의 감정 상태는? (불안? 급함? 호기심?)
- 표면적으로 원하는 것 말고, 진짜 원하는 것은?
- 여기 오기 전에 뭘 하고 있었고, 여기서 나간 후 뭘 할까?
- 결과를 혼자 볼까, 다른 사람에게 보여줄까?

**숨겨진 니즈 발굴:**
- 시나리오 A~D 외에, 이 앱을 쓸 만한 다른 상황이 있을까?
- 기존 도구에서 같은 작업을 할 때 사용자가 가장 짜증나는 점은?

**실수 방지 설계:**
- 입력에서 오류가 발생할 수 있는 곳은? (형식, 범위, 타입)
- 잘못된 값을 넣었을 때 어떻게 알려줄 것인가?
- 빈 상태에서 동작 시도 시 어떻게 처리할 것인가?

### Step 2: 디자인 리서치 + 기준서 작성 (`docs/02_디자인기준.md`)

**이 단계를 건너뛰고 코딩을 시작하면 안 된다.**
디자인 결정을 먼저 문서화한 후에만 Step 3으로 넘어갈 수 있다.

**디자인 리서치 실행:**

`/design-research` 스킬을 실행하라 (도메인: "[프로젝트명]").
스킬이 자동으로 최신 UI 검색 → 앵커 선택 → design-decisions.md 생성까지 처리한다.

스킬을 사용할 수 없으면 WebSearch로 직접 리서치하라:
- "[프로젝트 도메인] UI design 2025 2026"
- "[유사 서비스] web app modern design"
- "best [도메인] app design inspiration"

검색도 불가능하면, 아래 서비스 중 **프로젝트 성격에 맞는 것을 디자인 앵커로 선택**하라:

| 앵커 서비스 | 스타일 특징 | 적합한 상황 |
|------------|-----------|------------|
| Stripe Dashboard | 보라+흰, 카드 기반, 깔끔 | 금액/결제 |
| Notion | 따뜻한 화이트, 둥근 카드, 여백 | 문서/노트 |
| Linear | 미니멀 모노톤 | 생산성/태스크 |
| 토스/카카오뱅크 | 네이비+블루, 신뢰감 | 금융/계산 |
| Figma | 라이트 그레이, 좌우 패널 | 도구/에디터 |
| Vercel | 블랙+화이트, 극미니멀 | 개발자 도구 |

**리서치 후 결정할 것 (docs/02_디자인기준.md에 기록):**
- 참고한 서비스/사이트 (최소 2개, URL 포함)
- 컬러 팔레트: 메인, 서브, 배경, 강조 (hex 코드)
- 왜 이 디자인을 선택했는지 (1~2줄)
- 폰트, 컴포넌트 스타일 방향

**첫인상 설계:**
- 사용자가 앱을 열었을 때 첫 3초 안에 느껴야 할 감정은?
- "이거 진짜 서비스야?"라는 반응이 나오려면 어떤 요소가 필요한가?

**정보 구조 설계:**
- 숫자가 많은 화면에서 사용자가 길을 잃지 않으려면?
- 가장 중요한 정보를 어떻게 강조할 것인가?
- 부가 정보는 어떤 순서로 보여줄 것인가?

**초보자 배려 설계:**
- 전문 용어를 모르는 사람을 위한 설명 방식은?
- 각 입력 필드에 기본값과 안내를 어떻게 넣을 것인가?
- 모바일에서 엄지손가락 하나로 모든 조작이 가능한가?

### Step 3: 기능 + 기술 설계서 작성 (`docs/03_기능설계.md`)

Step 1, 2를 기반으로, 구현할 기능과 기술적 주의사항을 정리하라.

**AI가 흔히 놓치는 것들 (반드시 확인):**
- 금액/숫자 입력 시 천 단위 콤마 자동 포맷팅 + 콤마 포함 입력 허용
- 소수점/절사 처리가 필요한 곳은 Math.floor() 사용
- 인쇄 시 입력 패널이 출력되면 안 됨 (@media print)
- 이미지 업로드 후 새로고침해도 유지되는가? (localStorage base64)
- 빈 데이터/0 값일 때 오류 없이 처리되는가?
- 대량 데이터(항목 50개 이상) 시 렌더링이 느려지지 않는가?

**최신 웹 기술 리서치 (권장):**

구현 전에 WebSearch로 최신 CSS/JS 기능을 검색하여 적용 가능한 것이 있는지 확인하라:
- "modern CSS features 2025 2026" — container queries, :has(), color-mix() 등
- "modern JavaScript features 2025" — 최신 문법, 새 API
- "web API new features 2025" — View Transitions, Popover API, Dialog 등

적용 가능한 최신 기능이 있으면 `docs/03_기능설계.md`에 기록하고 구현에 반영하라.
단, 브라우저 호환성을 확인하고 (Chrome/Edge/Safari 최신 버전 기준) 안정적인 기능만 사용하라.

### Step 4: 위 3개 문서를 읽고 구현하라

- 3개 문서를 **전부 읽은 후** 코드를 작성하라
- `/frontend-design` 스킬이 있으면 실행하여 전문가 수준 프론트엔드를 구현하라
- 문서에 적은 내용이 코드에 빠짐없이 반영되었는지 확인하라
- 한 파일이 500줄을 넘지 않도록 분리하라 (HTML/CSS/JS)

## 4. 구현 기준

- **형태**: 웹앱. 서버/DB 없이 브라우저에서만 동작
- **언어**: 전체 UI 한글. 영어 메뉴/라벨 금지
- **첫 화면**: 빈 화면 금지. 기본값이 채워져 있어서 바로 결과가 보여야 한다
- **디자인 수준**: Step 2에서 리서치한 결과를 기반으로 전문 서비스 수준으로 구현
- **반응형**: 데스크탑과 모바일 모두 자연스럽게 동작

## 5. 디자인 규칙 (AI가 지켜야 할 UX 제약)

> 색상, 폰트, 스타일은 AI가 Step 2에서 리서치하여 결정한다.
> 아래는 디자인 선택이 아니라 **기능적으로 반드시 지켜야 할 UX 규칙**이다.
> 프로젝트에 해당하지 않는 항목은 삭제하라.

### 레이아웃 규칙

| 규칙 | 내용 | 이유 |
|------|------|------|
| [레이아웃 유형] | [예: 2패널 — 좌측 입력/우측 미리보기] | 핵심 UX |
| 모바일 전환 | 1024px 미만: [탭 전환/아코디언/스크롤] | 반응형 필수 |
| [미리보기 규칙] | [예: A4 비율, 종이 느낌 그림자] | [이유] |

### 숫자/데이터 표시 규칙 (해당 시)

| 규칙 | 내용 | 이유 |
|------|------|------|
| 정렬 | 금액/숫자는 우측 정렬 | 자릿수 비교 |
| 포맷 | 천 단위 콤마, tabular-nums | 가독성 |
| 위계 | 핵심 결과 > 중간 결과 > 세부 항목 순으로 크기 차등 | 정보 강조 |

### 접근성 규칙 (필수)

| 규칙 | 내용 | 이유 |
|------|------|------|
| 터치 영역 | 모바일 버튼/입력 최소 44x44px | 애플 HIG 기준 |
| 포커스 표시 | 입력 필드 포커스 시 시각적 변화 필수 | 접근성 |
| 대비 | 본문 텍스트와 배경의 명도 대비 4.5:1 이상 | WCAG AA |

### AI에게: 디자인 자유도

위 규칙만 지키면 나머지는 자유롭게 결정하라:
- 컬러 팔레트 (메인/서브/배경/강조)
- 폰트 크기와 굵기 체계
- 카드 모서리, 그림자, 간격
- 버튼 스타일, 아이콘 형태
- 전체적인 분위기 (미니멀/따뜻함/날카로움)

> **작성 팁**: "디자인을 자유롭게" = AI가 아무렇게나 만든다는 뜻이 아니다.
> Step 2에서 리서치하고 문서화한 디자인 결정에 따라 일관성 있게 구현해야 한다.

## 7. 비즈니스 로직 / 계산 규칙

> 프로젝트의 핵심 로직을 수식/코드로 정확하게 기술한다.

```
[계산 흐름 작성]

예시:
항목별 금액 = 수량 x 단가
소계 = 모든 항목 금액의 합
할인 후 = 소계 - 할인액
세금 = Math.floor(할인 후 x 세율)    ← 절사 규칙 명시
총액 = 할인 후 + 세금
```

## 9. 기본값

앱을 처음 열면 아래 샘플 데이터가 채워져 있어야 한다 (빈 화면 금지):

| 항목 | 기본값 |
|------|--------|
| [입력 항목 1] | [샘플 값] |
| [입력 항목 2] | [샘플 값] |
| [데이터 항목 1] | [샘플 값] |
| [데이터 항목 2] | [샘플 값] |

> **작성 팁**: 기본값은 사용자가 "아, 이렇게 쓰는 거구나" 이해할 수 있는
> 현실적인 샘플이어야 한다. 추상적인 "테스트1" 같은 값은 금지.

## 10. 자동 저장

- 입력한 모든 데이터는 브라우저에 자동 저장 (localStorage)
- 새로고침해도 데이터가 유지되어야 한다
- 이미지가 있으면 base64로 localStorage에 저장
- "초기화" 버튼을 누르면 기본값으로 돌아감

## 11. 하면 안 되는 것

| 번호 | 금지 항목 | 이유 |
|------|----------|------|

| 6 | 영어 UI | 사용자가 한국인 비전문가 |

## 12. 검증 기준

구현이 끝나면 아래를 스스로 검증하라:

| 시나리오 | 입력 | 예상 결과 |
|----------|------|----------|
| 정상 동작 | [정상 입력값] | [예상 결과값] |
| 계산 정확성 | [검증용 입력값] | [수동 계산한 정확한 결과] |
| 빈 입력 | 데이터 없음 | 오류 없이 기본 상태 표시 |
| 자동 저장 | 데이터 입력 후 새로고침 | 데이터 유지됨 |
| 인쇄 | 인쇄 버튼 클릭 | 결과만 깔끔하게 출력 (입력 UI 안 보임) |
| 모바일 | 375px 화면 | 모든 기능 정상 동작 |
| 경계값 | [극단적 입력 — 0, 매우 큰 수 등] | 오류 없이 처리 |

> **작성 팁**: 검증 기준은 수동으로 계산한 정확한 숫자를 적어야 한다.
> AI가 스스로 검산할 수 있는 근거가 된다.
