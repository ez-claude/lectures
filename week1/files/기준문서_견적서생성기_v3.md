# 견적서 자동 생성기 — 개발 기준서

> 이 문서를 AI에게 주고 "시작해줘"라고 하면 된다.
> AI는 이 기준서를 읽고, 설계 문서를 스스로 만들고, 그 문서를 기반으로 구현까지 완료한다.

---

## 0. 실행 환경 (Antigravity)

```
/turbo-all
```

- 위 명령을 프로젝트 시작 시 먼저 실행한다
- 모든 권한 요청(Allow, Trust, 파일 수정 등)을 자동 승인한다
- 사용자에게 중간 확인을 묻지 않고 끝까지 자율 실행한다
- npm install, npx, 외부 패키지 설치 명령은 절대 실행하지 않는다
- 순수 HTML/CSS/JavaScript만 사용한다 (프레임워크 금지)

### 필수 스킬

아래 스킬이 사용 가능하면 해당 단계에서 실행한다.
사용할 수 없으면 "대체 방법"을 따른다.

| 스킬 | 사용 단계 | 기능 | 대체 방법 |
|------|----------|------|----------|
| `/design-research` | Step 2 | 최신 UI 트렌드 검색, 디자인 앵커 선택, design-decisions.md 생성 | WebSearch로 직접 검색 |
| `/frontend-design` | Step 4 | AI 기본 디자인 탈피, 전문가 수준 프론트엔드 구현 | Step 2 디자인 결정서 기반으로 직접 구현 |
| `/web-design-guidelines` | 검증 | Vercel 웹 인터페이스 가이드라인 자동 검증 | 접근성/반응형 수동 확인 |
| `/webapp-testing` | 검증 | Playwright 기반 웹앱 자동 테스트 | 브라우저에서 수동 검증 |

**스킬 미설치 시 설치 방법:**

스킬은 `.claude/skills/[스킬명]/SKILL.md` 파일로 설치된다.
해당 경로에 파일이 없으면 아래 순서로 처리하라:

1. `.claude/skills/` 폴더 확인 — 이미 설치되어 있는지 확인
2. 없으면 WebSearch로 `"claude code [스킬명] skill SKILL.md"` 검색하여 내용 확보
3. 검색도 실패하면 "대체 방법"으로 수동 진행

## 1. 프로젝트 목표

소규모 사업자/프리랜서가 **엑셀 없이 전문적인 견적서를 3분 안에 작성하고 바로 출력**할 수 있는 도구를 만든다.

핵심 기준:
- 엑셀보다 빠르고 직관적이어야 한다
- 출력물이 실제 비즈니스에서 바로 쓸 수 있을 만큼 깔끔해야 한다
- 계산은 정확해야 한다 (부가세 절사 등)
- 디자인은 최신 B2B SaaS 수준이어야 한다 (AI가 직접 리서치하여 결정)

## 2. 사용자 시나리오

이 시나리오들을 분석하고, 각 시나리오를 해결하는 기능을 설계하라.

### 시나리오 A: 급한 견적
> "거래처에서 견적서 달라는데, 엑셀 파일 찾기 귀찮아요. 바로 입력해서 PDF로 보내고 싶어요."

사용자가 알고 싶은 것: 3분 안에 입력 → 미리보기 확인 → 인쇄/PDF

### 시나리오 B: 할인 적용
> "이 거래처는 오래된 곳이라 10% 깎아주기로 했는데, 부가세 계산이 헷갈려요."

사용자가 알고 싶은 것: 할인 적용 후 정확한 금액, 부가세 자동 계산

### 시나리오 C: 여러 버전
> "같은 품목인데 할인 있는 버전/없는 버전 두 개를 비교해서 보여주고 싶어요."

사용자가 알고 싶은 것: 빠르게 조건을 바꿔가며 비교

### 시나리오 D: 모바일
> "출장 중에 폰으로 견적서 수정해야 하는데, 기존 엑셀은 폰에서 안 돼요."

사용자가 알고 싶은 것: 모바일에서도 자연스럽게 입력/수정 가능한지

## 3. AI 실행 순서 (코드 작성 전 필수)

코드를 바로 작성하지 마라. 아래 4단계를 순서대로 실행하라.

### Step 1: 사용자 분석서 작성 (`docs/01_사용자분석.md`)

위 시나리오를 바탕으로, 아래에 대해 깊이 생각하고 정리하라.

**사용자가 진짜 원하는 것:**
- 이 앱을 여는 순간 사용자의 감정 상태는? (급함? 짜증? 기대?)
- 표면적으로 원하는 것(견적서 작성) 말고, 진짜 원하는 것은?
- 견적서를 만든 후 다음 행동은? (이메일 전송? 인쇄? 카톡?)
- 결과물을 누구에게 보여줄까? (거래처 담당자? 대표이사?)

**숨겨진 니즈 발굴:**
- 시나리오 A~D 외에, 이 앱을 쓸 만한 다른 상황이 있을까?
- 기존 도구(엑셀, 한글, 워드)에서 견적서를 만들 때 사용자가 가장 짜증나는 점은?
- 거래처마다 양식이 다를 수 있는데, 이걸 어떻게 해결할 것인가?

**실수 방지 설계:**
- 품목 입력에서 "단가에 콤마를 넣으면 오류"가 발생할 수 있는가?
- 할인율을 50%로 잘못 넣었을 때 어떻게 알려줄 것인가?
- 빈 견적서를 인쇄하려고 할 때?
- 사업자번호를 틀리게 입력했을 때?

### Step 2: 디자인 리서치 + 기준서 작성 (`docs/02_디자인기준.md`)

**이 단계를 건너뛰고 코딩을 시작하면 안 된다.**
디자인 결정을 먼저 문서화한 후에만 Step 3으로 넘어갈 수 있다.

**디자인 리서치 실행:**

`/design-research` 스킬을 실행하라. 이 스킬이 자동으로:
1. "견적서 생성기" 도메인에 맞는 최신 UI 트렌드를 검색하고
2. 적합한 디자인 앵커 서비스를 선택하고
3. 컬러/폰트/컴포넌트 스타일을 구체적으로 결정하고
4. `docs/design-decisions.md`를 생성한다

> 스킬이 설치되어 있지 않으면 아래 명령으로 설치 후 실행하라:
> ```
> 스킬 파일을 .claude/skills/design-research/SKILL.md 에 생성
> ```
> 또는 스킬 없이 직접 WebSearch로 아래를 검색하여 수동 리서치하라:
> - "invoice generator UI design 2025 2026"
> - "B2B SaaS form preview layout design"
> - "best invoice app design inspiration"

**첫인상 설계:**
- 사용자가 앱을 열었을 때 첫 3초 안에 느껴야 할 감정은? → "이거 전문 서비스 아니야?"
- 그 반응을 만들려면 어떤 색상/레이아웃/타이포가 필요한가?
- 즉시 미리보기에 샘플 견적서가 보여야 한다 — 빈 화면은 불안감을 준다

**정보 구조 설계:**
- 입력 패널의 섹션 순서는? (위에서 아래로 자연스럽게 채울 수 있는가?)
- A4 미리보기에서 가장 눈에 들어와야 하는 것은? (총액? 회사명?)
- 품목이 10개 이상일 때 레이아웃이 깨지지 않는가?
- 금액 숫자의 시각적 위계 — 소계 < 부가세 < 총액 순으로 강조

**초보자 배려 설계:**
- 사업자번호 형식(000-00-00000)을 모르는 사람을 위한 안내는?
- 각 입력 필드에 placeholder와 기본값을 어떻게 넣을 것인가?
- 모바일에서 테이블 입력이 가능한가? (엄지손가락으로 조작)
- "이 버튼을 누르면 어떻게 되는지" 망설이는 사용자를 위한 설계는?

### Step 3: 기능 + 기술 설계서 작성 (`docs/03_기능설계.md`)

Step 1, 2를 기반으로, 구현할 기능과 기술적 주의사항을 정리하라.

**AI가 흔히 놓치는 것들 (반드시 확인):**
- 금액 입력 시 천 단위 콤마 자동 포맷팅 + 콤마 포함 입력 허용
- 부가세 Math.floor() 절사 — 반올림 금지
- 인쇄 시 입력 패널이 출력되면 안 됨 (@media print)
- 로고/서명 이미지 업로드 후 새로고침해도 유지되는가? (localStorage base64)
- 품목 0개일 때 오류 없이 처리되는가?
- 모바일에서 품목 추가/삭제 버튼이 충분히 큰가? (최소 44x44px)

**최신 웹 기술 리서치 (권장):**

구현 전에 WebSearch로 최신 CSS/JS 기능을 검색하여 적용 가능한 것이 있는지 확인하라:
- "modern CSS features 2025 2026" — container queries, :has(), color-mix() 등
- "modern JavaScript features 2025" — 최신 문법, 새 API
- "web API new features 2025" — View Transitions, Popover API, Dialog 등

적용 가능한 최신 기능이 있으면 `docs/03_기능설계.md`에 기록하고 구현에 반영하라.
단, 브라우저 호환성을 확인하고 (Chrome/Edge/Safari 최신 버전 기준) 안정적인 기능만 사용하라.

### Step 4: 위 3개 문서를 읽고 구현하라

- 3개 문서를 **전부 읽은 후** 코드를 작성하라
- `/frontend-design` 스킬이 있으면 실행하여 전문가 수준 프론트엔드를 구현하라
- 문서에 적은 내용이 코드에 빠짐없이 반영되었는지 확인하라
- 한 파일이 500줄을 넘지 않도록 분리하라 (HTML/CSS/JS)

## 4. 구현 기준

- **형태**: 웹앱. 서버/DB 없이 브라우저에서만 동작
- **언어**: 전체 UI 한글. 영어 메뉴/라벨 금지
- **첫 화면**: 빈 화면 금지. 샘플 데이터가 채워져 있어서 바로 미리보기가 보여야 한다
- **디자인 수준**: Step 2에서 리서치한 결과를 기반으로 전문 비즈니스 앱 수준으로 구현
- **반응형**: 데스크탑과 모바일 모두 자연스럽게 동작

> 색상, 폰트, 스타일은 AI가 Step 2에서 리서치하여 결정한다.
> 아래는 디자인 선택이 아니라 **기능적으로 반드시 지켜야 할 UX 규칙**이다.

### 레이아웃 규칙

| 규칙 | 내용 | 이유 |
|------|------|------|
| 2패널 | 데스크탑: 좌측 입력(스크롤) / 우측 미리보기(고정) | 핵심 UX |

| 규칙 | 내용 | 이유 |

| 규칙 | 내용 | 이유 |
|------|------|------|

위 규칙만 지키면 나머지는 자유롭게 결정하라:
- 컬러 팔레트 (메인/서브/배경/강조)

- 전체적인 분위기 (미니멀/따뜻함/날카로움)

## 7. 계산 규칙

```
품목별 금액 = 수량 x 단가

항목별 할인 = 품목별 금액 x (항목 할인율 / 100)

소계 = 모든 (품목별 금액 - 항목별 할인)의 합

전체 할인 = 소계 x (전체 할인율 / 100), 원 미만 버림

할인 후 금액 = 소계 - 전체 할인

부가세(VAT) = 할인 후 금액 x 10%, 원 미만 버림 (Math.floor)

총액 = 할인 후 금액 + 부가세
```

### 반드시 지켜야 할 규칙

| 규칙 | 내용 |
|------|------|
| 부가세 절사 | 원 미만 버림(Math.floor). 반올림 금지 |
| 금액 표시 | 천 단위 콤마, 우측 정렬 |
| 할인 0%일 때 | 할인 행 표시하지 않음 |
| 금액 입력 | 콤마 포함 입력 허용 (1,000,000 → 1000000으로 파싱) |

## 8. A4 미리보기 구조

미리보기는 실제 인쇄될 모습과 동일해야 한다.

```
+--------------------------------------------+

## 11. 자동 저장

- 입력한 모든 데이터는 브라우저에 자동 저장 (localStorage)
- 새로고침해도 데이터가 유지되어야 한다
- 로고/서명 이미지도 base64로 localStorage에 저장
- "초기화" 버튼을 누르면 기본값으로 돌아감

## 13. 검증 기준

구현이 끝나면 아래를 스스로 검증하라:

| 시나리오 | 입력 | 예상 결과 |
|----------|------|----------|
| 기본 계산 | 웹개발 1,000,000 x 1 + 디자인 500,000 x 2 + 유지보수 300,000 x 3 | 소계 2,900,000 / 부가세 290,000 / 총액 3,190,000 |
| 절사 확인 | 단가 3원 x 수량 10 | 부가세 = Math.floor(30 x 0.1) = 3원 |
| 할인 계산 | 소계 1,000,000 + 전체할인 7% | 전체할인 70,000 / 할인후 930,000 / 부가세 93,000 / 총액 1,023,000 |
| 빈 품목 | 품목 0개 | 오류 없이 0원 표시 |
| 자동저장 | 데이터 입력 후 새로고침 | 데이터 유지됨 |
| 인쇄 | 인쇄 버튼 클릭 | 견적서만 A4에 맞게 출력 (입력 패널 안 보임) |
| 모바일 | 375px 화면 | 탭 전환으로 입력/미리보기 동작 |
| 콤마 입력 | 단가에 "1,500,000" 입력 | 정상 파싱, 오류 없음 |
