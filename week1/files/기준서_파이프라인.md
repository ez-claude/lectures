# AI 프로젝트 파이프라인 — 문서 체계

> 하나의 문서가 모든 걸 하지 않는다.
> 각 문서는 하나의 역할만 하고, 다음 문서의 입력을 만든다.

---

## 왜 나눠야 하나

AI에게 "기획하고 설계하고 만들고 검증해줘"를 한번에 시키면:
- 설치하다가 인증을 시도하고
- 디자인을 찾으러 브라우저를 열고
- 중간에 npm install을 반복하고
- 뭘 하고 있는지 AI도 사람도 모르게 됨

사람한테도 "기획서 쓰고 설계하고 코딩하고 테스트해"를 한번에 시키면 똑같이 꼬인다.

해결: **한 문서 = 한 역할. 순서대로 실행.**

---

## 파이프라인 구조

```
[Doc 1: 기획]        ← 사용자 아이디어를 정리
  "뭘 만들 건지"
  입력: 사용자의 한 줄 설명
  출력: 기능 목록, 대상, 제약사항
       ↓
[Doc 2: 설계+룰]     ← 기획을 기술 설계로 변환
  "어떻게 만들 건지"
  입력: Doc 1의 출력
  출력: 파일 구조, 컴포넌트 설계, 환경 규칙, 금지사항
       ↓
[Doc 3: 구현]        ← 설계대로 실제로 만듦
  "만들어"
  입력: Doc 2의 출력
  출력: 동작하는 앱
       ↓
[Doc 4: 검증+다듬기]  ← 만든 결과를 확인하고 다듬음
  "확인해, 고쳐"
  입력: Doc 3의 결과 + Doc 2의 규칙
  출력: 검증 완료된 앱
```

---

## 각 문서의 역할 (엄격하게)

### Doc 1: 기획서

**하는 일**: 뭘 만들 건지 정리
**하지 않는 일**: 기술 결정, 코드, 설치, 디자인

```
[이 문서의 역할]
사용자의 아이디어를 정리해서 기능 목록으로 만든다.

[사용자 입력]
"___을/를 만들고 싶어"

[AI가 할 일]
1. 프로젝트 유형 파악 (웹앱/문서/도구)
2. 핵심 기능 목록 뽑기 (5~10개)
3. 대상 사용자 정의
4. 제약사항 정리 (서버 없음, 로그인 없음 등)
5. 빠진 것 없는지 한번 점검
6. 사용자에게 확인: "이렇게 이해했는데 맞나요?"

[AI가 하면 안 되는 일]
- 코드 작성
- 패키지 설치
- 파일 생성
- 디자인 검색
- 기술 스택 결정

[출력 형식]
아래 형식으로 정리해서 보여줘:

프로젝트명: ___
목적: ___
대상: ___
핵심 기능:
  1. ___
  2. ___
  3. ___
  ...
제약사항:
  - 서버/DB 없음
  - 로그인 없음
  - ...
```

---

### Doc 2: 설계서 + 룰북

**하는 일**: 기획을 기술 설계로 변환. 환경 규칙 설정.
**하지 않는 일**: 코드 작성, 앱 실행

```
[이 문서의 역할]
Doc 1의 기획서를 받아서, AI가 코드를 만들 때 따라야 할 설계와 규칙을 만든다.

[입력]
Doc 1에서 만든 기획서를 여기에 붙여넣어줘.

[AI가 할 일]
아래 항목을 모두 포함하는 "구현 지시서"와 "검증 지시서"를 만들어줘.

[구현 지시서에 반드시 포함할 것]

1. 환경 제약 (최상단, 절대 규칙)
   - 사용할 프레임워크
   - 설치 금지 목록 (npm install 금지, 외부 패키지 금지 등)
   - CSS 방식 (순수 CSS만 등)
   - 파일 구조 방침

2. 기능별 구현 방법
   - 각 기능을 어떤 방식으로 구현할지
   - 차트 → SVG/CSS 직접
   - 인쇄 → window.print()
   - 파일 업로드 → FileReader API
   - (외부 패키지가 필요한 기능은 대체 방법을 명시)

3. 계산/처리 규칙
   - 계산식 (있으면)
   - 소수점/반올림 처리
   - 데이터 흐름

4. 디자인 방향
   - 구체적 색상 (hex 코드)
   - 레이아웃 구조
   - 반응형 기준점
   - 참고 스타일 (검색하지 말고, 텍스트로 설명)

5. 기본값
   - 앱 첫 실행 시 보여줄 샘플 데이터

6. 금지 목록
   - 설치 금지, 서버 금지, 로그인 금지 등 전부 나열

[검증 지시서에 반드시 포함할 것]

1. 체크리스트
   - 각 기능이 동작하는지 확인할 항목
   - 계산 정확도 검증 방법
   - 모바일 확인 항목

2. 알려진 위험
   - 이 프로젝트에서 AI가 실수할 수 있는 것

[AI가 하면 안 되는 일]
- 코드 작성
- 패키지 설치
- 앱 실행
- 웹사이트 방문/디자인 검색

[출력]
두 개의 문서를 만들어줘:
1. "구현 지시서" — Doc 3에서 사용
2. "검증 지시서" — Doc 4에서 사용
```

---

### Doc 3: 구현 지시서

**하는 일**: 설계대로 코드를 만들어서 동작하는 앱을 완성
**하지 않는 일**: 기획 변경, 디자인 검색, 배포, 검증

```
[이 문서의 역할]
Doc 2에서 만든 구현 지시서를 받아서, 그대로 코드를 만든다.

[입력]
Doc 2에서 만든 "구현 지시서"를 여기에 붙여넣어줘.

[AI가 할 일]
구현 지시서에 적힌 대로만 만들어.
- 환경 제약을 지켜
- 기능 목록을 빠짐없이 구현해
- 계산 규칙을 정확히 따라
- 디자인 방향을 반영해
- 기본값을 넣어
- 금지 목록을 지켜

다 만들면 "구현 완료. Preview에서 확인해주세요." 라고 알려줘.

[AI가 하면 안 되는 일]
- 구현 지시서에 없는 기능 추가
- npm install / 외부 패키지 설치 (지시서에서 금지했으면)
- 웹사이트 방문 / 디자인 검색
- 배포
- "이것도 추가하면 좋겠는데요" 같은 제안
```

---

### Doc 4: 검증 + 다듬기 지시서

**하는 일**: 만든 결과를 확인하고, 문제 있으면 수정
**하지 않는 일**: 새 기능 추가, 구조 변경

```
[이 문서의 역할]
Doc 2에서 만든 검증 지시서를 기준으로, Doc 3의 결과물을 확인한다.

[입력]
Doc 2에서 만든 "검증 지시서"를 여기에 붙여넣어줘.

[AI가 할 일]
검증 지시서의 체크리스트를 하나씩 확인해.
각 항목에 대해:
- 동작하면 → [OK]
- 안 되면 → [FAIL] + 원인 + 수정

모든 항목을 확인한 후 결과를 보여줘:

  [검증 결과]
  1. ___: [OK]
  2. ___: [OK]
  3. ___: [FAIL] → 원인: ___ → 수정 완료
  ...

  전체: N개 중 N개 통과

[디자인 다듬기]
검증 후, 디자인에서 개선할 부분이 있으면:
- 여백, 정렬, 색상 미세 조정
- 모바일에서 깨지는 부분 수정
- 사용하기 불편한 부분 개선

수정할 때 기존 동작을 깨뜨리지 마.

[AI가 하면 안 되는 일]
- 새 기능 추가
- 구조 변경
- 패키지 설치
```

## 실행 방법

### Antigravity에서 (실습용)

Doc 1~2는 ChatGPT/Claude 등 대화형 AI에서 진행하고,
Doc 3~4는 Antigravity에서 실행.

```
[Step 1] 아무 AI에게 Doc 1 붙여넣기 → "대출계산기 만들고 싶어"
         → 기획서 받음

[Step 2] 같은 AI에게 Doc 2 붙여넣기 → 기획서 함께 전달
         → 구현 지시서 + 검증 지시서 받음

[Step 3] Antigravity 새 프로젝트 → Doc 3 + 구현 지시서 붙여넣기
         → 앱 만들어짐

[Step 4] 같은 Antigravity 채팅에 Doc 4 + 검증 지시서 붙여넣기
         → 검증 + 다듬기
```

### Claude Code / Cursor 등에서 (고급)

서브에이전트로 병렬 처리 가능:
- Agent 1: Doc 2 (설계) 실행
- Agent 2: Doc 3 (구현) 실행 (Agent 1 완료 후)
- Agent 3: Doc 4 (검증) 실행 (Agent 2 완료 후)

---

## 핵심 원칙

1. **한 문서 = 한 역할** — 기획서가 코드를 만들지 않음. 구현서가 검증하지 않음.
2. **순차 실행** — 이전 문서의 출력이 다음 문서의 입력.
3. **금지사항 명시** — 각 문서마다 "하면 안 되는 일"을 명확히.
4. **사람은 중간중간 확인** — Doc 1 결과 확인 → Doc 2 결과 확인 → Doc 3 실행 → Doc 4 검증.

이 구조의 장점:
- 어디서 문제가 생겼는지 바로 알 수 있음 (Doc 3에서 꼬이면 구현 지시서 문제)
- AI가 범위를 벗어나지 않음 (각 문서가 범위를 제한)
- 문서 하나만 바꾸면 나머지는 그대로 재사용 가능
