3 sim: 장기 실패 시뮬레이션 (3단계)
가장 큰 장점: 아직 발생하지 않은 버그를 배포 전에 잡습니다. 실제로 터진 후 원인을 찾는 게 아니라, 코드가 완성된 시점에 미리 시나리오를 돌립니다. 코드를 잘 모르는 상태에서 AI에게 맡길 때 특히 유용합니다.
S1
실패 시뮬레이션 — 코드를 따라간다
변경된 파일을 실제로 Read한 뒤, "3개월 후 + 규모 10배" 상황을 설정하고 코드를 한 줄씩 따라가며 터지는 지점을 찾는다.

추적 대상: 하드코딩된 값 / 의존성 변화 / 규모 한계 (메모리, 처리 시간, 동시 접속) / 환경 차이 (OS, 인코딩, 권한)

규칙: "터질 수 있다" (추상적) 금지 → "이 파일 N번째 줄에서 이 입력이면 터진다" (구체적)
S2
근본 원인 탐색 — 최하위 레이어까지
S1에서 발견한 문제의 근본 원인을 파악하고, 같은 유형의 문제가 코드 어디에 더 숨어있는지 전체 탐색한다.

탐색 루프: 원인 후보 발견 → "이걸 고치려면 다른 걸 먼저 고쳐야 하는가?"
   • Yes: 아직 근본 원인 아님. 하위 레이어 코드/설정 Read 후 계속
   • No: 여기가 수정 위치. 탐색 종료

표면 원인에서 멈추지 않음. 수정 가능한 최하위 레이어까지 강제로 내려간다.
S3
수정 + 확인 시뮬레이션 (루프 최대 2회)
발견된 문제를 수정하고, 수정한 코드로 다시 S1 시뮬레이션을 1회 돌려 확인한다.

수정 위치 강제 검증: S2 근본 원인 위치 = S3 수정 위치 (불일치 시 수정 금지)
fallback 패턴 금지: if X is None: return default_X 형태 차단
→ 증상을 우회하는 게 아닌, 근본 원인 위치 직접 수정만 허용

2회 후에도 실패 시: AI가 혼자 계속 고치지 않음 → 사용자 승인 필요
